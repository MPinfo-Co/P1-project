# 規格文件使用說明

> 本文件說明 `Spec/` 目錄的結構與撰寫規範，供 SD 撰寫規格時參考。

---

## 目錄結構

```
Spec/
  {fn_功能}/
    Api/                      ← API 規格，每支 API 一個檔案
      {功能}_{動作}_api.md
    {功能}_00_overview.md     ← 功能總覽
    {功能}_01_{畫面}.md       ← 畫面規格
    {功能}_02_{畫面}.md
    …
```

---

## 命名規則

| 項目     | 規則                    | 範例                       |
| ------ | --------------------- | ------------------------ |
| 功能目錄   | `{功能編號}`              | `fn_user/`、`auth/`       |
| 功能總覽   | `{功能}_00_overview.md` | `fn_user_00_overview.md` |
| 畫面規格   | `{功能}_{序號}_{畫面簡述}.md` | `fn_user_01_list.md`     |
| API 規格 | `{功能}_{動作}_api.md`    | `fn_user_add_api.md`     |

**閱讀順序**：功能總覽 → 畫面規格 → API 規格

---

## 各檔案格式

### 功能總覽 `{功能}_00_overview.md`

**用途**：功能入口，供 SA/PM 快速 review 設計範圍。

**section 結構**：
- H1 = 功能標題，如 `# 帳號管理功能規格(fn_user)`
- H2 以下為內容區塊：
  1. `## 功能說明` — 功能描述 + 雛形畫面連結（格式：`雛形畫面：[▶ 開啟雛形](...)`，不含 section anchor）
  2. `## 主要使用角色` — 需要哪個權限才可進入
  3. `## 畫面清單` — 表格：畫面名稱 / 畫面規格 / 畫面關聯

---

### 畫面規格 `{功能}_NN_{畫面}.md`

**用途**：單一畫面的完整設計規格，PG 實作的直接依據。

**內容**：
- H1 = 畫面名稱（即為檔案主題），如 `# 查詢畫面`、`# 新增畫面/修改畫面`
- 雛形連結（`雛形畫面：[▶ 開啟雛形](../../Prototype/{頁面}.html#{section})`）
- 欄位說明（篩選列、清單表格 或 表單欄位）
- 操作說明（每個按鈕的行為，含 API 引用 + 畫面跳轉）

**表格欄標規範**：

| 表格類型 | 欄標                          |
| ---- | --------------------------- |
| 篩選列  | 欄位 / 輸入元件 / 說明              |
| 清單表格 | 欄位 / 顯示元件                   |
| 表單欄位 | 欄位 / 輸入元件 / 必填 / 驗證規則 / 預設值 |

**畫面邏輯差異不大時，可共用畫面規格**：例如，將新增/修改畫面規格合併，以 `# 新增畫面/修改畫面` 為主 section，處理邏輯差異以 `# 畫面差異` 表格補充。

---

### API 規格 `Api/{功能}_{動作}_api.md`

**用途**：後端 API 的完整規格，同時供前端了解 request/response 格式。

**section 結構**：
- H1 = 檔名加「規格」，如 `# fn_user_add_api規格`
- H2 以下為內容區塊：
  1. `## 介面說明` — 僅含 Method、Endpoint
  2. `## 傳入參數` — 以 `### 參數類型：Query Parameter / Path Parameter / Request Body` 次階標示，混合型可多段
  3. `## 處理邏輯` — 以 `### 檢核` / `### 執行` 次階區分
     - `### 檢核`：表格格式，欄標：檢核項目 / 失敗條件 / 回應訊息
     - `### 執行`：列表，描述通過檢核後的操作步驟
  4. `## 傳回結果` — 表格格式：Response / Message / data；4xx 統一填「依據檢核邏輯」

**Response Envelope**

所有回應統一格式：

```json
{ "message": "...", "data": ... }
```

- `message`：每個狀態碼皆帶，SD 在規格中指定文字內容
- `data`：有回傳資料時帶（如查詢結果），無資料時省略

**傳入參數表格欄標**：

| 參數類型           | 欄標                        |
| -------------- | ------------------------- |
| Query Parameter / Request Body | 欄位 / 型別 / 必填 |
| Path Parameter | 參數 / 型別        |

**傳回結果表格欄標**：Response / Message / data

**型別寫法**：`string`、`integer`、`boolean`、`object`、`array`；陣列元素型別已知時寫 `integer[]`、`string[]`、`object[]`

## 撰寫規範
### 一般

- **設計名詞**（欄位名稱、畫面名稱、按鈕文字）：一律用 `**粗體**`，全文用詞須一致
- **按鈕**：用 `[按鈕名稱]` 表示
- **流程轉換**（呼叫 API、切換畫面、跳頁）：用 `→` 表示
- **物件命名**（欄位、參數、傳入值）：直接對應畫面文字，不使用 id 命名；變數命名由 PG 決定
- **資料表名稱**：一律加 `tb_` 前綴，如 `tb_users`、`tb_roles`
- **失敗訊息**：不寫在畫面規格，由 API 規格的「傳回結果」負責

### API 引用格式

在畫面規格的「操作說明」中，以下格式引用 API：

```
**[按鈕名稱]**
- → `Api/{功能}_{動作}_api.md`
  - 傳入：欄位名稱（對應畫面文字）
  - 成功：描述後續行為（如：重新整理清單、→ 查詢畫面）
```

跨功能 API 路徑從 `Spec/` 根目錄起算：

```
- → `../framework/Api/framework_{動作}_api.md`
```
